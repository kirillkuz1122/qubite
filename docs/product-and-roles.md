# Продуктовая модель и роли

## Как сейчас понимать Qubite

Qubite — это рабочий прототип платформы для учебных соревнований. На текущем этапе ядром продукта являются турниры и путь участника через них.

Главная продуктовая ставка сейчас:

- пользователь должен попасть в турнир быстро и без путаницы;
- organizer должен суметь собрать рабочий турнир;
- служебные роли должны поддерживать платформу, но не являются центром UX.

## Главные сущности продукта

- `User` — аккаунт участника или служебной роли
- `Team` — команда для командных турниров
- `Task` — единица задания в банке задач
- `Tournament` — главный объект платформы
- `Roster entry` — список допуска / привязка участника или команды
- `Submission` — отправка ответа в runtime
- `Audit log` — журнал чувствительных действий

## Турнир как ядро продукта

В текущем проекте турнир — почти единственный главный объект системы. Вокруг него строятся:

- регистрация и вход;
- roster/code access;
- task runtime;
- leaderboard/results visibility;
- organizer-процессы;
- аналитика.

### Что подтверждается кодом

- форматы турнира: `individual`, `team`
- статусы хранения: `draft`, `published`, `ended`, `archived`
- вычисляемые lifecycle-состояния: `registration_open`, `registration_scheduled`, `starting_soon`, `live`, `live_late_join`, `ended`, `archived`
- режимы допуска: `open`, `registration`, `roster_only`, `code`
- режимы кодов: общий или персональный

## Роли

## `user`

Основной пользователь платформы.

Доступно:

- регистрация и вход;
- участие в турнирах;
- решение задач;
- просмотр рейтинга, результатов и части аналитики;
- создание команды и участие в командном контуре.

Это главный role-contour, вокруг которого должен строиться README и большая часть продуктовой документации.

## `organizer`

Роль человека, который собирает и запускает турниры.

Доступно:

- создание и редактирование турниров;
- работа с task bank;
- импорт задач и roster из Excel;
- просмотр результатов;
- управление собственным organizer-контуром.

Как роль получается:

- через organizer application;
- или вручную через административное назначение.

На текущем этапе organizer-функциональность уже есть, но UX здесь местами сырой.

## `moderator`

Служебная внутренняя роль контроля.

Доступно:

- работа с очередью модерации задач;
- разбор organizer applications;
- часть операций по контролю пользователей.

Это не основной продуктовый персонаж, а внутренний operational contour.

## `admin`

Сервисная и одновременно операционная роль управления платформой.

Доступно:

- управление пользователями, их ролями и статусами;
- удаление пользователей (аккаунты переходят в статус `deleted` на 7 дней для возможности восстановления, после чего окончательно стираются системой);
- удаление/правка отдельных сущностей платформы (турниры, задачи, команды);
- административные overview-экраны и детальная графическая аналитика (рост аудитории, просмотры, почта, ресурсы);
- аудит чувствительных действий в виде живой ленты (Real-time).

Admin — не просто “расширенный organizer”, а platform-level роль.

## `owner`

Защищённая над-роль над admin.

Особенности:

- owner недоступен для обычного web-demote/block flow;
- назначение и передача owner выполняются отдельно;
- web/API не должны относиться к owner как к обычному admin.

Эксклюзивные полномочия:

- доступ к блоку "Управление системой": глобальные рубильники для мгновенного отключения регистрации, рассылки писем, создания и участия в турнирах;
- управление "Режимом обслуживания" (Maintenance Mode) с секретным доступом через байпас-ссылки;
- окончательное удаление (Hard Delete) и восстановление пользователей из карантина.

В документации owner стоит описывать отдельно, а не сводить к “самый главный admin”.

## Как устроен task bank

Task bank сейчас правильнее описывать как `личный + общий`.

- organizer работает со своими задачами;
- существуют shared/published механики;
- есть moderation-status для задач;
- импорт задач через XLSX уже реализован.

## Командный режим

Командный режим есть в коде и работает, но в текущем проекте это второстепенный сценарий по сравнению с обычным participant-flow.

Поэтому в docs его стоит:

- описывать как реальную часть продукта;
- не выдавать за главный UX-путь платформы.

## Что уже есть, а что ещё нет

### Уже подтверждается кодом

- базовая auth-система;
- email verification flow и email-based challenge flows;
- турниры и task runtime;
- рейтинг и аналитические выборки;
- команды;
- organizer/moderation/admin/owner-контуры;
- импорт задач и roster из Excel;
- helper codes / code-entry сценарии.

### Пока не нужно выдавать за готовую продуктовую часть

- code runner для задач по программированию;
- AI-проверка как полноценный пользовательский сценарий;
- апелляции;
- visual/drag-and-drop task types;
- VK / Telegram login.

## Known issues для продуктового описания

- participant UX сильнее остальных контуров;
- organizer/moderation/admin UX пока заметно слабее;
- регистрация и verify-flow требуют усиления;
- логика рейтинга для пользователя пока недостаточно прозрачна;
- часть контуров выглядит как рабочий прототип, а не отполированная продуктовая система.
